Automatización

Cron + n8n + Eventos

El sistema se auto-gestiona: límites, limpieza, notificaciones, reportes.

⏰ Cron Jobs

Polling de tareas * * * * *

Reset límites mensuales 0 0 1 * *
Reintentar fallidas */30 * * * *
Reporte semanal 0 9 * * 1

Datos

Esquema de Base de Datos

4 tablas nuevas para el Studio. Cada tarea, consumo y artista queda registrado.

studio_tasks

id UUID · user_id INT · artist_id VARCHAR
task_type VARCHAR(32) · seedance_task_id VARCHAR(64)
status VARCHAR(16) · credits INT · prompt TEXT
result_url TEXT · processing_time INT
created_at · completed_at

studio_usage

id UUID · user_id INT · month DATE
reels_generated INT · credits_consumed INT
UNIQUE(user_id, month)

studio_artists

id UUID · artist_id VARCHAR(8) FK
character_refs JSONB — URLs de fotos de referencia
created_at · updated_at

studio_templates

id UUID · name VARCHAR(128) · prompt TEXT
category VARCHAR(64) · artist_id VARCHAR
is_public BOOLEAN · created_by INT

Los videos generados expiran en el CDN de Seedance. El webhook handler los descarga automáticamente al storage local (OVH / R2) y actualiza result_url con una URL permanente.

Roadmap · 15 días

Fases incrementales. Cada fase produce algo testeable en Docker.

1
Fundación
Días 1-2
  • Mock de Seedance (Flask)
  • SDK base con types
  • Docker compose con mock
  • Test: curl → mock → respuesta

API + Worker
Días 3-7
  • Studio API (POST /studio/generate)
  • Studio Worker (cola Redis)
  • Polling de tareas en progreso
  • Webhook handler + storage
  • Test: API → Worker → Mock → Webhook
3
Integración
Días 8-12
  • Frontend: Reel Studio en dashboard

Producción

  • Test E2E completo
