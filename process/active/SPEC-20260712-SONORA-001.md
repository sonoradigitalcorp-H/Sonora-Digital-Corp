## 1. Objetivo

Establecer la infraestructura de datos multi-tenant (Supabase + Hasura + PostgreSQL) con bot central de Telegram como canal de entrega, schema completo de monetización, RAG por cliente en Open Notebook, y pipeline de scraping con Playwright — todo orquestado por LangChain + LangGraph.

## 2. Value Driver

- **Automatización**: eliminar JSON files → base de datos real con GraphQL
- **Retención**: gamificación + saludos personalizados fidelizan fans
- **Revenue**: monetización híbrida ($BEAT + Stripe) desde el día 1
- **Fundador-independencia**: el sistema se auto-alimenta via Playwright scraping headless
- **Streaming**: WebSockets para dashboards en tiempo real

## 3. Functional Requirements

| ID | Descripción |
|----|-------------|
| FR1 | Supabase project con Auth (Google OAuth + email) y JWT configurado |
| FR2 | Hasura conectado a Supabase PostgreSQL con JWT verification |
| FR3 | Schema PostgreSQL completo: tenants, users, telegram, token_ledger, greetings, transactions, quests, schedules, auto_messages, content_generations, scraped_metrics, knowledge_bases |

## 6. Edge Cases

|----|-------------|
| EC1 | Bot recibe mensaje de usuario no registrado → responde con enlace de registro |

## 7. Technical Approach

```
Capa 0 - Infraestructura (Docker):
  - Supabase: proyecto cloud (auth, storage, edge functions)
  - Hasura: docker container con HASURA_GRAPHQL_JWT_SECRET = Supabase JWT secret
  - PostgreSQL: Supabase managed (no Docker local)
  - Redis: docker container (colas + pub/sub)
  - Qdrant: docker container (vectores)
  - Neo4j: docker container (grafos)
  - Open Notebook: docker container (RAG)
  - Playwright: script Python con cron (headless)

Capa 1 - Schema (PostgreSQL via Hasura):
  - infra/migrations/010_sonora_os.sql
  - Tablas: tenants, users, telegram_users, token_ledger, greetings, transactions,
            quests, quest_completions, rewards, schedules, auto_messages,
            notifications, content_generations, scraped_metrics, knowledge_bases
  - RLS policies por tenant_id en cada tabla

Capa 2 - Bot Telegram:
  - apps/sonora_bot/main.py
  - python-telegram-bot v20+
  - Solo handlers: message_handler, command_handler(/start)
  - Sin lógica: escribe a Redis "telegram:inbox", lee de "telegram:outbox"
  - Docker container aparte

Capa 3 - LangChain Engine:
  - apps/sonora_engine/main.py (FastAPI)
  - Router Agent (LangGraph) con nodos:
    tenant_lookup → intent_classifier → route
  - Sub-agentes: chat, monetization, gamification, automation, knowledge
  - Tools: Hasura MCP, Stripe MCP, FAL MCP, OmniVoice MCP, Neo4j MCP, Qdrant MCP

Capa 4 - Scraping Pipeline:
  - apps/sonora_scraper/playwright_scraper.py
  - Cron: 6:00 AM daily
  - Headless Chromium + Playwright
  - Logging + retry + alert on failure

Capa 5 - WebSocket Server:
  - apps/sonora_engine/websocket.py
  - FastAPI WebSocket + Redis Pub/Sub
  - /ws/{tenant_id} endpoint
  - Eventos: agent:*, scraping:*, payment:*, greeting:*
```

## 9. Events to Emit

| Evento | Trigger | Payload |
|--------|---------|---------|
| `telegram:message:received` | Bot recibe mensaje | `{chat_id, text, user_id}` |
| `agent:routing:completed` | Router clasifica intento | `{session_id, intent, agent}` |
| `payment:stripe:completed` | Stripe checkout success | `{transaction_id, amount, tenant_id}` |
| `payment:beat:burned` | Fan gasta $BEAT | `{user_id, amount, reason}` |
| `greeting:requested` | Fan pide saludo | `{greeting_id, artist_id, fan_id}` |
| `greeting:approved` | Artista aprueba saludo | `{greeting_id, audio_url}` |
| `scraping:completed` | Playwright termina extracción | `{artist_id, metrics}` |
| `scraping:failed` | Playwright falla | `{artist_id, error}` |
| `quest:completed` | Fan termina quest | `{user_id, quest_id, reward}` |
| `gamification:levelup` | Fan sube de nivel | `{user_id, new_level}` |
| `websocket:connected` | Dashboard se conecta | `{tenant_id, session_id}` |

## 10. Kill Criteria

- Supabase proyecto no se puede crear → abortar
- Hasura JWT verification falla por más de 2 días → abortar
- Tests no pasan en verde en 3 intentos → abortar
- Schema migration rompe datos existentes → rollback

## 11. Scale Criteria

- >1000 mensajes/día en Telegram → escalar bot a worker pool
- >100 clientes → Hasura Cloud (managed) en vez de self-hosted
- >1M transacciones/día → particionar token_ledger por mes
- Scraping >10 artistas → paralelizar con asyncio
