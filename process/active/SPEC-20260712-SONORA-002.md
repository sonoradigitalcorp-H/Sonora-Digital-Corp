## 1. Objetivo

Implementar gamificación (Play/Work/Learn to Earn), monetización híbrida ($BEAT + Stripe), y dashboard en vivo con WebSockets sobre la fundación Sonora OS.

## 2. Value Driver

- **Revenue**: Stripe + $BEAT burns generan ingresos directos
- **Retention**: gamificación aumenta engagement y visitas diarias
- **Automation**: pagos y rewards 100% automáticos
- **Visibility**: dashboard en vivo permite al cliente ver métricas en tiempo real

## 3. Functional Requirements

| ID | Descripción |
|----|-------------|
| FR1 | Quests configurables por cliente (plantilla + extensiones), categorías Play/Work/Learn |
| FR2 | Sistema de niveles: Bronze → Silver → Gold → Platinum con beneficios por nivel |
| FR3 | Rewards en $BEAT desde pool fijo del cliente, con burns al gastar |
| FR4 | Stripe Checkout para pagos fiat (saludos, contenido) |
| FR5 | $BEAT ledger: earn, burn, transfer. Pool fijo total (1M/5M). 50% burn por transacción |
| FR6 | Saludos personalizados híbridos: IA genera → artista aprueba → entrega |
| FR7 | Dashboard REST endpoints: revenue, streams, quests, token circulation, greetings |
| FR8 | WebSocket streaming de eventos en tiempo real para dashboard |
| FR9 | Leaderboard por cliente (top fans por XP/$BEAT) |
| FR10 | Tests unitarios + integración en verde |

## 6. Edge Cases

| ID | Descripción |
|----|-------------|
| EC1 | Pool $BEAT agotado → rewards se pausan, se notifica al admin |
| EC2 | Stripe webhook falla → reintento con idempotency key |
| EC3 | Artista rechaza saludo → fan recibe reembolso en $BEAT o fiat |
| EC4 | Quest diaria expira a medianoche → se reinicia automáticamente |
| EC5 | Leaderboard empate → desempate por logros adicionales |
| EC6 | WebSocket desconectado → cliente reconecta con último estado conocido |

## 7. Technical Approach

```
Capa - Gamification:
  apps/sonora_engine/gamification.py:
    - QuestEngine: validar, completar, calcular rewards
    - LevelSystem: XP → nivel, beneficios
    - Leaderboard: top fans por tenant

Capa - Payments:
  apps/sonora_engine/payments.py:
    - StripeCheckout: crear sesión, webhook handler
    - BEATLedger: earn, burn, transfer, balance
    - GreetingHandler: flujo híbrido IA+artista

Capa - Dashboard:
  apps/sonora_engine/dashboard.py:
    - Revenue stats, stream counts, token circulation
    - Quest progress, greeting status, leaderboard
    - WebSocket event emitter mejorado
```

## 8. Dependencies

| Dependencia | Propósito |
|-------------|-----------|
| stripe | Payments fiat |
| httpx | Llamadas a Stripe API |
| redis | WebSocket pub/sub, rate limiting |

## 9. Events

| Evento | Trigger |
|--------|---------|
| `gamification:quest:completed` | Fan completa quest |
| `gamification:levelup` | Fan sube de nivel |
| `payment:stripe:checkout:created` | Nueva sesión Stripe |
| `payment:stripe:completed` | Pago exitoso |
| `payment:beat:earned` | Fan gana $BEAT |
| `payment:beat:burned` | Fan gasta $BEAT |
| `greeting:requested` | Fan pide saludo |
| `greeting:approved` | Artista aprueba |
| `greeting:rejected` | Artista rechaza |
| `dashboard:refresh` | Datos del dashboard actualizados |

## 10. Kill Criteria

- Stripe API no disponible por >1 hora → pausar payments fiat
- Pool $BEAT se agota sin reposición → pausar rewards
- Tests no pasan en verde → no deploy

## 11. Scale Criteria

- >1000 quests/día → cachear leaderboard en Redis
- >100 pagos/día → webhook worker pool
- >50 clientes → particionar dashboard queries por tenant
