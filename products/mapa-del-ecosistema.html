El Modelo Mental IMPORTANTE

Este sistema no es una aplicacion. Es un ecosistema de 7 capas apiladas. Cada capa depende de la de abajo. Cuando tocas algo en la capa 2 (infraestructura), puede romper algo en la capa 6 (automation). Eso NO es culpa del modelo LLM — es arquitectura de software normal.

La regla de oro: si arreglas algo y se rompe otra cosa, es porque las capas estan acopladas. La solucion no es un modelo mas grande — es modularizar.

Preguntas Frecuentes

Por que cuando arreglo algo se rompe otra cosa?

Por las dependencias entre capas. Ejemplo real de hoy:

1. Neo4j estaba configurado con HEAP=2G + CACHE=1G = 3GB
2. La maquina tiene 3.2GB RAM total
3. n8n intentaba arrancar pero el OOM killer lo mataba (exit 137)
4. El frontend tambien fallaba por falta de recursos DNS (timeout por swap)
5. Solucion: reducir Neo4j a 1.5GB -> n8n arranca, frontend arranca, swap baja
Moraleja: No es que "todo este desarreglado". Es que un bottleneck (RAM) cascada a todos los servicios de arriba.

Por que el contexto no se ve completo cuando pido ayuda?

Porque el sistema tiene 11,735 lineas de Python + configs + scripts + 97 skills + 9 containers. Ningun modelo LLM puede procesar TODO eso en una sola llamada.

La solucion no es un modelo mas grande. DeepSeek V4 Flash tiene contexto de 128K tokens — eso son ~90,000 palabras, suficiente para unos 3,000-4,000 lineas de codigo. El problema es que nuestro sistema es 3x mas grande que eso.

Que hacer: En vez de pedir "mira todo el contexto", hay que:
1. Especificar QUÉ capa/componente necesitas revisar
2. Usar las skills (workflow-enforcer, ceo-workflow) que ya tienen el contexto precargado
3. Aceptar que un sistema complejo se arregla por partes, no todo a la vez

Que es RAG exactamente? Donde esta?

RAG = Retrieval Augmented Generation. Busca informacion relevante ANTES de que el LLM responda.

El pipeline es:
Texto → Ollama (nomic-embed-text) → Vector (768 numeros) → Qdrant (almacena) → Busqueda semantica → LLM responde

Donde esta: src/core/rag.py (191 lines) + src/core/embeddings.py + Qdrant container + Ollama service
Estado actual: 171 puntos con embeddings reales. Busqueda semantica funcionando.
Cuando falla: Si Ollama no corre (no hay embeddings) o Qdrant esta caido (no hay donde buscar).

Que es MCP? Para que sirve?

MCP = Model Context Protocol. Es un standard abierto (como USB para LLMs) para que el modelo pueda usar herramientas.

Sin MCP: el LLM solo puede generar texto.
Con MCP: el LLM puede leer archivos, ejecutar comandos, consultar bases de datos, etc.

Donde esta: Servidor MCP en Docker (puerto 8000), definido en docker/mcp-server/
Tools disponibles: 10 tools (read_file, write_file, search_code, run_command, query_neo4j, etc.)
Diferencia con RAG: RAG es para BUSCAR informacion. MCP es para EJECUTAR acciones.

Que es el Agent Harness?

El orquestador (src/core/orchestrator.py) que decide QUE agente debe manejar cada peticion.

Tiene 11 agentes: hermes, research, code, memory, gbrain, openclaw, pr, y mas.
Cada agente es un archivo en src/core/agents/. Heredan de AgentBase.
El orquestador recibe un mensaje, evalua que agente es el mejor, y le pasa el control.
Estado: 166 lines, funciona bien, pero los 17 archivos de agentes son casi identicos (boilerplate).

Te parece si pongo Kimi 2.7 o DeepSeek Pro para orquestar?

Si, pero con estas condiciones:

1. Usalo SOLO para sesiones de orquestacion/arquitectura/planificacion — NO para el dia a dia (coding, debugging)
2. El costo es 10-20x mayor. DeepSeek Flash nos cuesta ~$0.14/1M. Pro seria ~$2-4/1M. Para una sesion de 2 horas hablando con el asistente, serian ~$0.50 con Flash vs ~$5-10 con Pro.
3. El verdadero cuello de botella sigue siendo la modularizacion (4/10), no el modelo.
4. Modelo recomendado para orquestacion: DeepSeek V4 Pro (mejor relacion calidad-precio para tareas complejas). Kimi K2.5 es segunda opcion.
5. NO es necesario para arreglar los problemas actuales (API keys, modularizacion, cobertura de tests).

Conclusion sobre el modelo

DeepSeek V4 Flash es suficiente. Si quieres experimentar con Pro o Kimi para orquestacion pesada, adelante — pero no esperes que eso arregle los problemas estructurales. Lo que realmente necesita el sistema es:

✅ Rotar 29 API keys (seguridad)
✅ Refactorizar god modules (modularizacion)
✅ Subir cobertura a 80% (calidad)
✅ Deployar landing pages y vender (negocio)

Ningun modelo LLM hace eso por si solo. Se necesita ingenieria de software classica.
