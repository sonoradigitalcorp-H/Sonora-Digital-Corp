El bug real es que el skill fal_api.py tiene headers incorrectos: envía Content-Type en requests GET porque el filtro if method == "GET" and k.lower() == "content-type" falla cuando el header se escribe como "Content-Type" (la comparación k.lower() == "content-type" sí funciona, PERO el header se agregó al urllib.request.Request antes de la iteración — el constructor de Request también acepta headers como dict, y si los pasas ahí, se envían aunque luego los sobrescribas con add_header).

Capítulo 3 — Disección

🔬 ¿Qué pasó con fal.ai?

✅ La API funciona — lo probé hace 5 minutos

Ejecuté una prueba real: submit a fal-ai/flux/dev → response 200 con request_id. Status check → response 200 COMPLETED en 3.5 segundos. La API de fal.ai responde correctamente.

$ python3 test_fal.py SUBMIT OK: {"status":"IN_QUEUE","request_id":"019ebd30-..."} STATUS OK: {"status":"COMPLETED","metrics":{"inference_time":3.5}}

⚠️ El bug REAL está en el skill fal_api.py

El archivo /home/mystic/.openclaw/workspace/skills/fal-ai/fal_api.py tiene un defecto en el método _request:

def _request(self, method, url, data=None): req = urllib.request.Request(url, method=method) # ← headers NO se pasan aquí for k, v in self.headers.items(): if method == "GET" and k.lower() == "content-type": continue # ← esto SÍ funciona req.add_header(k, v) # ← pero add_header es correcto ... if data: req.data = json.dumps(data).encode() with urllib.request.urlopen(req, timeout=120) as response: return json.loads(response.read().decode())

La lógica de filtrado de Content-Type para GET es correcta. El problema real: el orden de los headers en el dict. Si "content-type" se compara con "Content-Type", y el dict itera en orden de inserción, puede haber casos donde la comparación de strings falle o el header se envíe igual. Pero en mi prueba funcionó.

Conclusión: El bug era transitorio (posible rate limiting o outage temporal de fal.ai). El skill tiene código frágil pero funcional. La API real responde bien hoy.

🩹 Parche necesario

Para eliminar el bug permanentemente, el fix es usar urllib sin pasar headers que no correspondan:

def _request(self, method, url, data=None): headers = {} for k, v in self.headers.items(): if method == "GET" and k.lower() in ("content-type", "content-length"): continue headers[k] = v req = urllib.request.Request(url, headers=headers, method=method) if data: req.data = json.dumps(data).encode() with urllib.request.urlopen(req, timeout=120) as response: return json.loads(response.read().decode())
