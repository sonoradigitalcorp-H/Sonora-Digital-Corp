# evolution-auto — Automatic Pattern Detection, Proposal Generation & Evolution Loop

**Template**: SKILL-TEMPLATE.md v1.0.0
**Version**: 1.0.0
**Audit ID**: SKILL-EA-001

---

## 1. Business Objective

Continuously detect patterns in session data, generate improvement proposals, and auto-apply low-risk fixes to drive the system toward higher enterprise scores without human intervention.

## 2. Inputs (Gherkin)

```gherkin
Given scored sessions exist in the Learner
And the Learner has extracted learning metrics
When the Auto-Evolve cycle is triggered
```

## 3. Outputs (Gherkin)

```gherkin
Then patterns are detected from session data (FREQUENT_ERROR, SKILL_GAP, REGRESSION, OPPORTUNITY, TIME_CORRELATION, SUCCESS_PATTERN)
And actionable proposals are generated from detected patterns
And low-risk proposals are auto-applied without human review
And an evolution report is generated and persisted
```

## 4. Events

```
Events:
- evolution:cycle:started: fired when an evolution cycle begins
- evolution:cycle:completed: fired when a cycle finishes (count of patterns + proposals)
- evolution:pattern:detected: fired when a new pattern is found (pattern type + confidence)
- evolution:proposal:generated: fired when a proposal is created (proposal type + risk)
- evolution:proposal:applied: fired when a low-risk proposal is auto-applied (proposal_id)
- evolution:daily:report: fired after the daily cycle completes
```

```
Dependencies:

```
Tools:

## 7. Policies

```
Policies:
- Only low-risk proposals are auto-applied; medium/high-risk proposals require human review
- Proposals must follow ADR format from process/templates/ADR.md
- Patterns with confidence < 0.3 are discarded
- Sessions must be scored before pattern detection runs
- Each evolution cycle produces exactly one report
- The evolution loop must not modify session data directly
```

## 8. Success Metrics

```gherkin
Success Metrics:
- cycle_completion: Given an AutoEvolve cycle When it finishes Then all stages complete without error
  Target: 100% of cycles complete
- pattern_recall: Given known session patterns When PatternDetector runs Then it finds them
  Target: >= 80% recall on synthetic test data
- proposal_quality: Given a detected pattern When ProposalGenerator runs Then it produces a valid ADR
  Target: 100% of proposals have all required fields
- auto_apply_rate: Given low-risk proposals When auto_apply runs Then they are applied
  Target: 100% of low-risk proposals are auto-applied
```

## 9. Failure Conditions

```
Failure Conditions:
- Session store empty: no sessions to analyze (not a failure, empty state)
- Pattern file corrupt: JSON decode error on patterns.json (reset to empty)
- Enterprise score unavailable: computation fails (cycle continues without it)
- Proposal save fails: permission error writing to process/active/proposals/ (log and skip)
- Systemd timer miss: daily cycle skipped (next run catches up)
```

## 10. Recovery Procedure

```
Recovery Procedure:
1. If session store is empty: report 0 patterns, 0 proposals — no action needed
2. If patterns.json is corrupt: delete the file, start fresh
3. If enterprise score fails: log warning, continue cycle without score
4. If proposal save fails: log error, retry once, skip if still failing
5. If systemd timer missed: run manually with `python3 -m evolution.auto_evolve --once`
6. Report all failures to state/logs/skills/evolution-auto.log
```
